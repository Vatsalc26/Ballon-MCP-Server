# Contributor Starters

This page exists to help new contributors find bounded, useful Balloon work quickly.

## Best Starter Labels

If you are opening or triaging issues, use these first:

1. `good first issue`
2. `help wanted`
3. `docs`
4. `benchmark`
5. `host-ux`

## Good First Issue Shapes

The best first Balloon issues are:

1. clearly bounded
2. easy to verify
3. tied to context fidelity, benchmark repeatability, or host clarity
4. unlikely to require changing the whole architecture

## Recommended Starter Issues

### Docs and onboarding

1. tighten one confusing install or demo step in a real host flow
2. improve one benchmark worksheet or scorecard instruction
3. improve one public example request file

### Benchmark work

1. rerun one four-lane scenario and record the outputs cleanly
2. score one SCBench starter-suite checkpoint sequence
3. improve benchmark wording so outputs are easier to compare

### Host UX

1. verify one host path in VS Code, Cline, or Roo and file a compatibility report
2. tighten one fallback path when prompt routing is flaky
3. improve one host-specific doc from a real rerun

### Quality and guardrails

1. improve repaired wording without dropping protected areas or verification obligations
2. improve staged wording while keeping bounded-next-step quality explicit
3. tighten verification so a known regression becomes easier to catch

## Not Good First Issues

Avoid giving new contributors issues that:

1. redefine Balloon's core identity
2. require broad architecture changes across many files
3. make provider-side claims outside honest MCP boundaries
4. depend on hidden local setup that strangers cannot reproduce

## Recommended Maintainer Formula

When you write a starter issue, include:

1. the user-visible problem
2. the smallest useful fix
3. acceptance criteria
4. how to verify it
5. whether it changes docs, runtime, or benchmark artifacts

## Good Starter Backlog Right Now

1. score the SCBench starter suite across `baseline / deterministic / assist / staged`
2. improve one host quickstart after a real rerun
3. tighten one public example so a stranger can copy it without guessing
4. add one regression check for a benchmark wording bug
5. capture one clean benchmark screenshot or GIF and document how it was produced
